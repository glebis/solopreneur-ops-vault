**Suggested comment:**
> "Cobus's architecture breakdown is exactly right from the technical side — and there's a non-developer orchestration question worth adding, because it's where the architecture meets a different class of practitioner.
>
> For a developer, the feedback loop when an agent team fails is rich: logs, traces, error messages, observable state at each node. You can see where coordination broke and why.
>
> For a non-developer orchestrating the same team, that feedback loop is opaque. The agent team returns output — or it doesn't. When it doesn't, the practitioner sees the final state but not the failure point. 'It didn't work' is not a debugging signal.
>
> What makes agent teams usable for non-developer orchestrators is designing for observable failure: each agent in the team should produce an intermediate output the orchestrator can inspect before the next agent runs. Not a log — a legible checkpoint. 'I found 5 sources. Here they are.' → review → 'I drafted a summary.' → review → 'Formatted and ready.' Done.
>
> The agent team then isn't a black box — it's a pipeline with human inspection points. Slower than a fully autonomous team, but the pattern that produces trust for practitioners who aren't comfortable debugging code.
>
> The architecture question for non-developer orchestration isn't 'how do I build the team?' It's 'where do I put the checkpoints that let me course-correct without reading logs?' The answer to that question is different for every workflow — but the principle is consistent: legible intermediate outputs over silent execution."

**Post URL:** [Cobus Greyling — Claude Code Agent Teams](https://www.linkedin.com/pulse/claude-code-agentteams-cobus-greyling-smnke) — developer commentary is strong and technical; the non-developer orchestration gap is entirely unfilled; Greyling's audience skews developer but includes technical-adjacent practitioners who are exactly the people your comment serves.

**Why relevant:** You live inside this problem. Your Obsidian vault is the memory layer for your Claude Code workflows, and you've had to solve the memory quality problem in practice: what to store, how to structure it, how to keep it current enough that agent output stays reliable. The insight you can add is the two-layer memory architecture — evergreen vs. operational — and the maintenance rhythm that prevents the agent from confidently citing stale information.

**Suggested comment:**
> "The architecture described here works — and the memory quality problem is worth naming, because it's what determines whether a memory-augmented agent becomes more useful over time or less.
>
> A Claude Code agent with access to a memory store will query it confidently regardless of whether the stored information is current. This is a feature when the memory is fresh and a serious problem when it's not: the agent cites a decision you reversed three months ago, a workflow that's changed, a priority that's been superseded. It doesn't know what it doesn't know.
>
> What actually makes a memory store produce reliable agent behaviour over time is a two-layer architecture with different update rhythms.
>
> Evergreen layer: your principles, frameworks, preferences, how you think about recurring problems — slow to change, updated quarterly at most. Highly reliable context for the agent. Store this in a dedicated section of your notes that you treat as stable.
>
> Operational layer: current projects, live decisions, this-week priorities — fast-changing, needs a weekly pass. One 'still current' tag per note, once a week. Any note that doesn't get that tag starts to be treated as potentially stale.
>
> The agent should treat these differently: evergreen notes as reliable context, operational notes as information that benefits from a freshness check before the agent acts on them.
>
> The technical setup described in the post is right. The maintenance contract that keeps it reliable over time is the thing most practitioners discover only after their agent confidently acts on something they changed months ago."

**Suggested comment:**
> "Greg's framing of the two poles is right — and there's a third path worth naming that sits between them and gets overlooked because it doesn't have a clean category name.
>
> No-code agents: visual, fast to build, ceiling set by the vendor's feature roadmap. You can build quickly but you can't go beyond what the platform supports.
>
> Coded agents: maximum flexibility, full control, ceiling is what you can build — but requires development skill and ongoing maintenance capacity.
>
> The middle path: structured prompting with maintained skill files. This is what tools like Claude Code's skills system enable — text-based agent configurations you write directly, more flexible than no-code platforms (because you're writing the instructions, not working around a visual interface), but without requiring you to write code (because the agent runtime handles execution).
>
> This path is accessible to non-developers who are comfortable with text editors and can write clear, testable, structured instructions. The ceiling isn't the platform vendor or your coding ability — it's what you can express clearly in language.
>
> The skill this path requires is different from both drag-and-drop and coding: it's the ability to specify what an agent should do, what inputs it needs, what the output should look like, and what 'wrong' means in a way that's testable. That's a learnable skill for most practitioners, and it's worth naming separately because it often gets people further than either pole — without the platform constraints or the development overhead."
